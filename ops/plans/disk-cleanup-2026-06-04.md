# Plan: Освобождение места на сервере (48G / 59G — 85%)

## Принципы безопасности
- Не трогать запущенные контейнеры и их данные
- Каждое действие — верифицировать до и после
- Пошагово, с проверкой после каждого шага
- Не трогаем containerd (14G) — это runtime-хранилище Docker, удаление сломает все контейнеры

---

## Фаза 1: Безопасная чистка (~4.3G)

### 1.1 Docker container logs — truncate (3.7G освобождения)
**Риск:** нулевой. Логи — не данные. Контейнеры продолжат писать в те же файлы.

- Truncate логи всех контейнеров через `truncate -s 0`
- Самые тяжёлые: supabase-kong (2.6G), supabase-auth (602M), shadowbox (297M)
- После: настроить log rotation в daemon.json (`max-size: 10m, max-file: 3`)

### 1.2 Ротация /var/log (~400M освобождения)
**Риск:** нулевой. Ротированные файлы — старые логи.

- `truncate -s 0 /var/log/btmp.1` (218M — лог неудачных входов)
- `truncate -s 0 /var/log/auth.log.1` (45M)
- Удалить старые ротации: `syslog.1`, `kern.log.1`, `ufw.log.1`
- Настроить logrotate для btmp (сейчас растёт бесконтрольно)

### 1.3 Удаление старого ядра (~300M освобождения)
**Риск:** минимальный. Текущее ядро 6.8.0-107, старое 6.8.0-106.

- Верифицировать: `uname -r` — убедиться что загружено 107
- `apt remove linux-image-6.8.0-106-generic`

## Фаза 2: Требует проверки (~1.8G потенциально)

### 2.1 nvm Node v24 у n8nuser (1.3G)
**Риск:** средний. Если n8n или другие сервисы n8nuser используют v24 — сломается.

- Проверить: что запускается из-под n8nuser и какую версию Node использует
- Если v24 не используется — удалить `/home/n8nuser/.nvm/versions/node/v24.13.0`

### 2.2 Supabase .git (542M)
**Риск:** низкий. .git нужен только для `git pull` обновлений self-hosted supabase.

- Проверить: как обновляется supabase (docker-compose pull или git pull?)
- Если обновление через docker images — .git не нужен, можно удалить
- Если через git pull — оставить, или запомнить что придётся re-clone при обновлении

---

## Фаза 3: Docker оптимизация (размер зависит от dangling)

### 3.1 Docker image prune
**Риск:** минимальный. Удаляет только untagged/dangling слои.

- `docker image prune -f` — удалит dangling images (не используемые контейнерами)
- Не трогает образы привязанные к запущенным контейнерам

### 3.2 Проверка containerd overlay
**Риск:** ВЫСОКИЙ. Не трогать без глубокого анализа.

- 11G overlayfs в containerd — это те же слои что Docker использует
- Не является «лишним» — это рабочее хранилище
- Возможная оптимизация: migration на Docker с fuse-overlayfs + cleanup, но это отдельная задача
