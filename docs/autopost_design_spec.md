# Техническое задание: Интерактивный Telegram-автопостер (BullRun)

Документ объединяет все обсужденные идеи по созданию бота для автопостинга в два канала (публичный и закрытый) с управлением очередью и предложкой целиком внутри Telegram, без привлечения веб-интерфейса и без скачивания медиафайлов на внешние сервера.

---

## 1. Архитектура и База Данных

Для реализации идеи используются две таблицы в Supabase: `autopost_bots` (настройки ботов) и `autopost_items` (очередь постов).

### Таблица `autopost_bots` (Настройки бота)
* `id` (UUID, primary key)
* `owner_id` (UUID, связь с пользователем сайта)
* `bot_token` (Text, токен от @BotFather)
* `username` (Text, юзернейм бота)
* `admin_tg_id` (BigInt, Telegram ID администратора — заполняется автоматически при старте)
* `is_active` (Boolean, статус работы)
* `active_mode_user_id` (JSONB, хранит текущий режим загрузки для админов, например: `{"12345678": "public_channel_id"}`)

### Таблица `channels` (Подключенные каналы)
* `id` (UUID, primary key)
* `autopost_bot_id` (UUID, внешняя ссылка)
* `tg_chat_id` (BigInt, ID канала/группы в Telegram)
* `title` (Text, название)
* `posts_per_day` (Integer, частота постов в день)
* `posting_times` (Text[], массив времени публикации, например: `["10:00", "18:00"]`)
* `buttons_config` (JSONB, настройки инлайн-кнопок под каждым постом в этом канале)

### Таблица `autopost_items` (Очередь контента)
* `id` (UUID, primary key)
* `bot_id` (UUID)
* `target_channel_id` (BigInt, ID канала-назначения)
* `file_ids` (Text[], массив ID файлов Telegram — для поддержки и одиночных фото, и альбомов)
* `caption` (Text, текст подписи)
* `status` (Text, статусы: `suggested` (предложка), `queued` (в очереди), `scheduled` (запланирован), `posted` (опубликован), `editing` (в процессе редактирования))
* `sort_order` (Integer, порядок сортировки)
* `scheduled_at` (Timestamp, время публикации)
* `posted_at` (Timestamp, фактическое время публикации)

---

## 2. Сценарий запуска и Onboarding (Связка с сайтом)

1. **Добавление токена:** Админ вводит только токен бота на сайте. Поле `admin_tg_id` изначально пустое.
2. **Активация в Telegram:** Бот запускается на сервере. Админ переходит в чат бота и пишет `/start`.
3. **Определение роли (Интерактивная привязка):**
   * Бот проверяет: если `admin_tg_id` в БД равен `null`, он выводит кнопку: `[ ✅ Я администратор ]`.
   * При клике бот записывает `ctx.from.id` в поле `admin_tg_id` бота.
   * Сайт по поллингу видит привязку админа и переводит интерфейс на шаг «Подключение каналов».
   * Все последующие пользователи, нажавшие `/start`, автоматически определяются как **гости** (им показывается интерфейс предложки).
4. **Подключение каналов:** Админ добавляет бота администратором в свои каналы (публичный и закрытый). Бот ловит событие `my_chat_member` и автоматически сохраняет каналы в БД.

## 3. Логика работы бота в Telegram (Интерфейс Админа)

Админ общается с ботом с помощью постоянной клавиатуры (Reply Keyboard) внизу экрана:

```
┌──────────────────────────────────────────┐
│ 🔄 Направление: Публичный 📢             │
├──────────────────────────────────────────┤
│ ➕ Добавить пост    │ 📋 Очередь         │
├──────────────────────────────────────────┤
│ 📥 Предложки (5)                         │
└──────────────────────────────────────────┘
```

### 1. Кнопка «🔄 Направление» (Переключатель режимов)
При клике переключает режим отправки для конкретного админа. 
* Если выбран `Публичный 📢` — все последующие присланные фото пойдут в очередь публичного канала.
* Если выбран `Приватный 🔒` — в очередь приватного канала.

### 2. Кнопка «➕ Добавить пост» (Прием картинок и альбомов)
* **Одиночное фото:** Сразу улетает в базу со статусом `queued` и привязкой к текущему каналу-назначению.
* **Альбом (несколько фото разом):** Бот ждет 1 секунду (собирает все фото с одним `media_group_id`). Затем присылает сообщение:
  > 📸 **Обнаружен альбом из X картинок!**
  > `[ 📦 Опубликовать альбомом ]` `[ ✂️ Разбить на X постов ]`
  * Выбор «Альбомом» -> создает одну запись в БД с массивом `file_ids`.
  * Выбор «Разбить» -> создает X отдельных записей в БД с одиночными `file_id`.

### 3. Кнопка «📋 Очередь» (Управление контент-планом)
Бот спрашивает, какую очередь показать: `[ 📢 Публичную ]` или `[ 🔒 Приватную ]`. Выводит ленту сообщений (до 15-20 последних постов). Под каждой картинкой в ленте находятся кнопки:
* `[ ⚡️ Опубликовать ]` — моментальный пост в канал, запись удаляется из очереди чата.
* `[ 📝 Изменить текст ]` — переводит бота в режим ожидания текста для этого поста. После ввода редактирует подпись под картинкой прямо в истории чата (`editMessageCaption`).
* `[ 🔒 В Приватный ]` (или `[ 📢 В Публичный ]`) — переносит пост в другую очередь. Пост исчезает из текущей ленты.
* `[ ❌ Удалить ]` — стирает пост из БД и удаляет само сообщение из чата Telegram.

### 4. Кнопка «📥 Предложки» (Модерация)
Бот присылает ленту присланных гостями постов с кнопками:
`[ ⚡️ Опубликовать сейчас ]` `[ 📥 В общую очередь ]` `[ ❌ Отклонить ]`

## 4. Логические дыры и их решения (Аудит дизайна)

Во время проектирования были выявлены и устранены следующие логические противоречия:

### 🔍 Дыра 1: Отправка фото до привязки каналов
* **Проблема:** Админ привязал свой аккаунт, но еще не добавил бота в каналы. Если он начнет слать картинки, бот не поймет, к какому `target_channel_id` их привязать.
* **Решение:** Если в таблице `channels` для этого бота нет ни одной записи, кнопка режима показывает `[ ⚠️ Нет подключенных каналов ]`. При попытке прислать фото бот отвечает: *«Сначала добавьте меня в канал как администратора!»*.

### 🔍 Дыра 2: Разделение подписей при разбиении альбома
* **Проблема:** Если админ присылает альбом с одной подписью и выбирает «Разбить на отдельные посты», подпись по умолчанию получит только первый пост.
* **Решение:** Бот разбивает альбом и спрашивает: *«Альбом разделен на X постов. Продублировать подпись на каждый пост?»* с кнопками `[ ✅ Да, продублировать ]` и `[ ❌ Оставить только на первом ]`.

### 🔍 Дыра 3: Конфликт планировщика при редактировании
* **Проблема:** Что если Cron-задача сработает ровно в тот момент, когда админ нажал «Изменить текст» и пишет новую подпись? Пост уйдет в канал недоделанным.
* **Решение:** При нажатии кнопки «Изменить текст» статус поста в БД временно меняется на `editing`. Планировщик (Cron) игнорирует любые посты в этом статусе. Как только админ присылает текст, статус возвращается в `scheduled`.

### 🔍 Дыра 4: Дыры в расписании при ручной публикации («схлопывание»)
* **Проблема:** Админ полистал очередь и опубликовал пост №3 «вне очереди» прямо сейчас. В расписании будущего образуется пустой слот.
* **Решение (Авто-решедулинг):** При любом ручном выкладывании или удалении поста из очереди, бот берет все оставшиеся запланированные посты (`status = 'scheduled'`), сбрасывает их в `queued` и заново прогоняет через планировщик. Они автоматически сдвигаются назад, заполняя пустоту, и плотно прижимаются к текущему моменту времени.

### 🔍 Дыра 5: Как гости предлагают новости в разные группы через одного бота
* **Проблема:** Бот один. Как понять, в какую предложку (для открытого или закрытого канала) гость прислал пост?
* **Решение (Deep-linking):** Под постами в публичном канале кнопка «Предложить новость» ведет на `t.me/bot?start=suggest_pub`, а в приватном — на `t.me/bot?start=suggest_priv`. При переходе бот считывает параметры и автоматически маркирует присланный контент нужным типом очереди.

## 5. Поддержка нескольких администраторов (Multi-Admin)

Проект изначально проектируется с учетом будущего масштабирования на работу нескольких администраторов с одним ботом:

### 1. Архитектурное решение в БД:
* Столбец `admin_tg_id` в таблице `autopost_bots` при расширении заменяется на массив `admin_tg_ids bigint[]` (или выносится в отдельную связующую таблицу `autopost_bot_admins` со столбцами `bot_id` и `admin_tg_id`).
* В коде бота проверка доступа меняется с `ctx.from.id === admin_tg_id` на проверку вхождения: `admin_tg_ids.includes(ctx.from.id)`.

### 2. Управление через сайт (Веб-панель):
* В веб-панели `/app/autopost` добавляется блок **«Администраторы бота»**.
* Владелец бота видит список всех привязанных админов (Telegram ID и их юзернеймы).
* Владелец может:
  * **Добавить админа вручную:** вписать Telegram User ID второго человека.
  * **Сгенерировать инвайт-ссылку:** бот генерирует специальный линк типа `t.me/bot?start=add_admin_SECRETCODE`. Второй админ переходит по ней, бот проверяет секретный код, сохраняет его ID в БД и присылает уведомление владельцу о новом админе.
  * **Отозвать права:** владелец в один клик может удалить любого администратора из списка на сайте, после чего бот мгновенно перестанет реагировать на команды этого человека.

### 3. Изоляция сессий (Отсутствие конфликтов):
* Текущий активный режим публикации (`Публичный` / `Приватный`) для каждого администратора хранится изолированно (в столбце `active_mode_user_id` таблицы `autopost_bots` в формате JSONB: `{ "id_админа_1": "public", "id_админа_2": "private" }`).
* Благодаря этому админы могут наполнять разные каналы одновременно с разных устройств, и их действия не будут сбивать настройки друг друга.

## 6. Автопринятие предложек (Auto-Accept)

Для каналов с проверенным, лояльным сообществом реализуется опция автоматической публикации предложенного пользователями контента без ручной модерации админом:

### 1. Настройки в БД и на Сайте:
* В таблицу `channels` добавляется флаг `auto_accept_suggestions` (Boolean, по умолчанию `false`).
* В веб-панели настроек для каждого канала добавляется переключатель (Toggle): **«Автопринятие предложений»**.
  * Если выключен — предложки висят в `📥 Предложки` и ждут ручного одобрения.
  * Если включен — предложенный контент сразу идет в работу.

### 2. Сценарий работы при включенном автопринятии:
1. Подписчик переходит по кнопке под постом в канал предложки (`t.me/bot?start=suggest_pub`).
2. Отправляет боту картинку с подписью.
3. Бот проверяет флаг `auto_accept_suggestions` для этого канала:
   * **Пост автоматически одобряется:** бот сразу создает запись в БД со статусом `queued` (в очередь) минуя статус `suggested`.
   * **Авто-планирование:** бот сразу запускает метод авто-решедулинга. Картинка автоматически занимает ближайший свободный слот контент-плана (например, через 2 дня) без какого-либо участия администратора.
4. Бот присылает подписчику теплое сообщение-подтверждение:
   > *«🎉 Спасибо! Ваш пост принят и автоматически запланирован к публикации на [Дата] в [Время].»*

### 3. Безопасность и защита от спама:
* Если какой-то пользователь начинает спамить нежелательным контентом, владелец может в любой момент выключить тумблер **«Автопринятие предложений»** на сайте. Бот мгновенно перейдет в режим ручной модерации, и все новые предложки снова будут ждать ручного одобрения админа.
