# Customers Actions Menu - 2026-04-24

## Goal

Перенести ручные операторские действия из старых экранов `CRM` и `Access` в основную таблицу `/app/customers`, чтобы админ работал из одного места:

- продлить доступ на фиксированный срок;
- выдать доступ навсегда;
- удалить человека из группы;
- сразу видеть правильный хвост после действия.

## What Exists Now

Готовая механика уже есть, но разбросана:

- `CRM`:
  - `+5 дней`
  - `+30 дней`
  - `Кик`
- `Access`:
  - `+5 дней`
  - `+30 дней`
  - `Кик`
- backend:
  - `POST /api/userbot/crm/subscribers/batch-add-days`
  - `POST /api/userbot/crm/subscribers/batch-kick`

Текущее поведение:

- batch extend обновляет `subscriptions.expires_at` и `subscriptions.status = 'active'`;
- batch kick пишет `access_events.event_type = 'kicked'`, `event_source = 'manual_batch'`, обновляет `subscriptions.last_access_event = 'kicked'` и `access_note`.

## Product Interpretation

Пользователь хочет не просто кнопку `+30`, а нормальное меню действий у строки клиента:

- `Продлить на 5 дней`
- `Продлить на 30 дней`
- `Выдать навсегда`
- `Удалить из группы`

И после действия человек должен попадать в понятное состояние:

- после продления: `Активный доступ`;
- после выдачи навсегда: `Активный доступ`;
- после ручного удаления: отдельный операторский хвост, а не просто потеряться в `Доступ закончился`.

## Main Decision

Для `Customers` нужен один компактный dropdown в колонке `Действия`, а не набор отдельных inline-кнопок.

Причина:

- таблица уже перегружена статусами и причинами;
- действия отличаются по риску;
- одно меню проще масштабировать, чем ряд кнопок.

## Scope

### 1. Frontend

В `CustomersPage.jsx`:

- добавить кнопку `Действия` у строки;
- раскрывать меню с действиями:
  - `Продлить на 5 дней`
  - `Продлить на 30 дней`
  - `Выдать навсегда`
  - `Удалить из группы`
- во время запроса блокировать повторный клик по этой строке;
- после успешного действия перезагружать workbench.

### 2. Backend

Переиспользовать существующие ручки, где это безопасно:

- `batch-add-days` использовать для:
  - `5 дней`
  - `30 дней`
  - `Навсегда` через `days = 'forever'`
- `batch-kick` использовать для удаления из группы.

Но нужно доработать семантику ручного удаления:

- сейчас `batch-kick` пишет общий `event_source = 'manual_batch'`;
- для `Customers` лучше различать ручное админское удаление как отдельный источник/причину;
- UI должен уметь отличать:
  - авто-кик;
  - ручной кик админом;
  - обычное истечение доступа.

### 3. Workbench Classification

Нужно расширить сегментацию `/api/customers/workbench`.

Сейчас есть:

- `customers-active`
- `customers-expired`
- `access`

Нужно добавить/явно поддержать:

- `removed-admin` или эквивалентный сегмент `Удален админом`

Правило:

- если `last_access_event = 'kicked'` и источник ручной, это не просто `Доступ закончился`;
- это отдельный операторский хвост.

## Proposed UX

### Tabs

Текущие основные вкладки:

- `Нажал старт`
- `Смотрели тарифы`
- `Не смогли оплатить`
- `Активный доступ`
- `Доступ закончился`
- `Не смог войти`

Добавить:

- `Удален админом`

Почему отдельная вкладка лучше, чем смешивать с `Доступ закончился`:

- ручное решение админа и естественное истечение доступа — разные состояния;
- по удаленным нужен отдельный возвратный сценарий;
- это снижает путаницу в таблице и в рассылках.

### Actions Menu

В строке клиента:

- одна кнопка `Действия`
- внутри меню:
  - `Продлить на 5 дней`
  - `Продлить на 30 дней`
  - `Выдать навсегда`
  - разделитель
  - `Удалить из группы`

Для `Удалить из группы`:

- обязательно `confirm`
- текст подтверждения без воды, с указанием клиента и канала, если они есть.

## Data Rules

### Extend

После продления:

- `subscriptions.status = 'active'`
- `expires_at`:
  - смещается на `+5` или `+30`
  - или становится `null` для `forever`
- человек должен перейти в `Активный доступ`

### Kick

После ручного удаления:

- пишется `access_events.event_type = 'kicked'`
- нужен понятный `event_source`, например `customers_manual_action`
- `subscriptions.last_access_event = 'kicked'`
- `subscriptions.access_note` должен явно говорить, что удаление сделал админ вручную из Customers
- человек должен перейти во вкладку `Удален админом`

## Open Question Resolved For MVP

Не делать пока сложные права/причины удаления.

Для MVP достаточно:

- одно действие `Удалить из группы`
- отдельный хвост `Удален админом`
- корректная запись причины и источника

- `Customers` шлет `action_source = customers` в batch-kick, а backend пишет отдельную семантику `customers_manual_admin_removed`.

- `Dossier` уже читает этот источник как отдельный manual-admin-removed сценарий.
